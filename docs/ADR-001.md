# ADR-001: Session-Based Quiz Architecture

## Status

Accepted

## Context

The Finance Quiz Platform will support multiple clients, including:

* Telegram Bot
* Web Application
* Future Mobile Application
* Public REST API

The application must allow users to:

* Start a quiz
* Resume an unfinished quiz
* Submit answers
* Track scores
* View leaderboards
* Support multiple quiz categories in the future

The architecture should avoid duplicating business logic across different clients.

---

## Decision

The application will be designed around the concept of a **Quiz Session**.

A quiz session represents a single attempt by a user to complete a quiz.

Neither the Telegram bot nor the website will contain quiz business logic. Both clients will communicate with the same backend services.

The service layer will expose operations such as:

* Start a quiz session
* Retrieve the current question
* Submit an answer
* Advance to the next question
* Complete a session
* Calculate the final score

---

## Domain Model

```
User
│
├── QuizSession
│      │
│      ├── QuizAttempt
│      │
│      └── Quiz
│
└── Statistics
```

Core database tables:

* users
* quizzes
* questions
* quiz_sessions
* quiz_attempts

Leaderboards will not be stored as a separate table. They will be generated dynamically from quiz session data.

---

## Architecture

```
Telegram Bot          Website
      │                 │
      └────────┬────────┘
               │
           FastAPI API
               │
         Service Layer
               │
      Repository Layer
               │
          SQLAlchemy ORM
               │
            SQLite / PostgreSQL
```

All clients use the same service layer.

Business rules exist only once.

---

## Benefits

* Single source of truth for quiz logic.
* Easy to add new clients (mobile app, Discord bot, Slack bot, etc.).
* Supports resume functionality.
* Enables analytics based on quiz attempts.
* Simplifies leaderboard generation.
* Encourages separation of concerns.
* Easier testing by isolating business logic from transport layers.

---

## Consequences

The project will follow the following layered architecture:

```
API
    ↓
Services
    ↓
Repositories
    ↓
Database
```

The Telegram bot and website will act only as presentation layers and will not directly access the database.

Future features such as daily quizzes, multiple categories, timed quizzes, achievements, and user statistics can be implemented without changing the overall architecture.
